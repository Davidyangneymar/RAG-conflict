# P2 Deliverable — Hand-off Notes

Version: `src.p2.__version__ = "0.1.0"`  (2026-04-18)

This file is the **cover letter** for the P2 package. It lists what is
in the box, what's required to run it, what the acceptance criteria
are, and how to reach the next module.

---

## 1. What you are receiving

The P2 module does two jobs on top of P1's JSON output:

1. **Stance analysis** — for every candidate pair emitted by P1, predict
   `support / oppose / neutral / filtered` using the FNC-1-trained BERT
   model, then fuse with the P1 NLI label into a single
   `agreement_signal`.
2. **Conflict typing** — classify every pair into one of seven conflict
   types (`none`, `hard_contradiction`, `temporal_conflict`,
   `opinion_conflict`, `ambiguity`, `misinformation`, `noise`) and
   attach a resolution policy the downstream generation layer can
   switch on.

P2 now also includes a **P6-facing prompt strategy layer**:

3. **Controlled-generation planning** — build `AnswerContext` from
   conflict-typed outputs, route policy-specific prompt strategy, produce
   two-stage prompts (analysis draft + constrained final answer), and
   apply abstention gate rules.

Output is a single JSON file, shape frozen in
[docs/CONTRACTS.md](docs/CONTRACTS.md).

## 4. Acceptance checklist (what the teammate should verify)

```bash
cd <unpacked_folder>
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

bash scripts/run_all_tests.sh
```

Expected output from the validator:

- **[1/5]** adapter contract: `All samples OK.`
- **[2/5]** fusion tests: `All 8 tests passed.`
- **[3/5]** conflict typing tests: `All 22 tests passed.`
- **[4/5]** prompt strategy tests: `All 5 tests passed.`
- **[5/5]** AVeriTeC closed loop: `Processed 4 sample(s).` followed by
  per-sample `type_counts` and a sanity table.

If any stage fails, the script exits non-zero — read the failure
message, it will name the failing file and assertion.

## 5. How downstream modules plug in

### 5.1 From a Python API

```python
from src.p2 import run_full_p2_pipeline_from_path

output = run_full_p2_pipeline_from_path("path/to/p1_output.json")
for sample in output.samples:
    for pair in sample.pair_results:
        if pair.conflict_type == "temporal_conflict":
            # your "prefer newest" branch
            ...
        elif pair.conflict_type == "ambiguity":
            # your "disambiguate first" branch
            ...
```

If downstream needs P6 answer plans directly:

```python
from src.p2 import run_full_p2_with_answer_plans_from_path

typed_output, answer_plans = run_full_p2_with_answer_plans_from_path(
    "path/to/p1_output.json"
)
for plan in answer_plans:
    prompt_for_stage_a = plan.prompt_bundle.stage_a_analysis_prompt
    prompt_for_stage_b = plan.prompt_bundle.stage_b_answer_prompt
```

### 5.2 From a JSON hand-off

```bash
python scripts/run_averitec_pipeline.py path/to/averitec.json \
    --out_dir outputs/my_run
# downstream reads outputs/my_run/p2_final.json
```

The JSON shape is documented in [docs/CONTRACTS.md](docs/CONTRACTS.md).
`conflict_type` and `resolution_policy` are the two fields a resolution
router should branch on.

## 6. What is out of scope for this deliverable

- **Real NLI**: P2 does not run an NLI model. It consumes P1's NLI
  label. When NLI is missing (e.g. AVeriTeC smoke runs), the fusion
  layer falls back to stance-only and marks the result in `notes`.
- **Entity disambiguation**: same-name entities are surfaced as
  `conflict_type="ambiguity"` but P2 does not try to resolve them. The
  project introduction assigns this to the retrieval / P3 layer.
- **Answer generation**: P2 hands `resolution_policy` to the
  controlled-generation module; we do not generate text ourselves.
- **Benchmarking metrics** (conflict-F1, stance-F1, abstention
  precision): P5's territory. P2 plumbs `gold_verdict` through
  `TypedSample` so P5 can read it.

## 7. Contact / next steps

- Contract changes must go through `docs/CONTRACTS.md` first — the
  vocabulary for `conflict_type` and `resolution_policy` is frozen;
  extend by adding values, not by renaming.
- If the downstream module needs a new field on `TypedPair`, open a
  PR against the P2 package; do not monkey-patch.
- When P1 ships real JSON, drop
  `src/p2/datasets/averitec.py` from the import path — P2's core
  does not depend on it.
